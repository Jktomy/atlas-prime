# Prime Spear Repository Path Scope

This contract controls repository-path eligibility for Athena's direct Spear route.

A Jayson-authorized direct Spear Weave may declare any safe repository-relative path in `Jktomy/atlas-prime`. Governance, lifecycle, schemas, workflows, Quest source, recovery source, generated contracts, and `tools/thread-engine/**` do not force Athena onto a different authoring route merely because of their location.

For direct Spear only, this contract supersedes two older path-gating rules while preserving every unrelated control in their source files:

- the self-change machine value and protected/self-change routing rows in `governance/athena-execution-route-contract.md`;
- the protected and self-change Change Routes bullets in `governance/source-lifecycle.md`.

Those older clauses remain historical compatibility evidence but do not control new direct-Spear transactions after this contract becomes canonical.

The direct Spear CLI opens repository-wide path scope only around one exact compile. Outside that invocation, existing path classification remains unchanged. The resulting compiler-bound mission remains tied to its exact base, declared paths, source blobs, payload hashes, candidate tree, final path set, one branch, one commit, one draft pull request, and independent readback.

Repository location is classification, not permanence authority. Private content remains outside Prime. Unsafe paths, stale inputs, undeclared files, automatic permanence, settings changes, and standing authority remain rejected. High-impact changes still receive their required validation and review. A Thread Engine self-change changes only a draft candidate and cannot make itself canonical.

Generated projections remain non-governing and follow their applicable deterministic generation contract. Path eligibility alone never converts generated bytes into authored truth.

The implementation must prove that one direct Spear compile can target governance source and `tools/thread-engine/**`, that the exact compiler-bound mission validates without a path-class detour, and that the scoped exception closes after compilation.
