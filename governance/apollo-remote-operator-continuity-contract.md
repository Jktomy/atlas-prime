# Apollo remote operator continuity

Apollo is the preferred normally available Windows work surface for native VS
Code, Windows-specific Atlas/Codex workflows, the first Glass Codex graphical
client, and the interactive Helios Control Deck. It is an operator endpoint,
not an indispensable server, source of truth, credential authority, monitoring
system, recovery controller, or persistent automation host.

The normative public-clean readiness record is
`governance/apollo-remote-operator-continuity-v1.json`, validated by
`schemas/apollo-remote-operator-continuity-v1.schema.json`.

## Endpoint and access roles

- Apollo hosts interactive native Windows work only.
- Hermes retains an independent portable Prime, GitHub, Element, and approved
  private-administration route plus clean-clone Prime recovery.
- Iris and Zeus are nonblocking viewing, communication, and human-approval
  companions; neither is the sole approval, administration, or recovery route.
- Iris, Hermes, and Zeus may reach only an explicitly approved Apollo surface
  through a separately governed private route. Public exposure and Microsoft
  Remote Tunnels are not required or implied.

Private, engineering, and Worldhopper environments have separate credentials,
workspaces, processes, extension storage, caches, cleanup, and revocation. A VS
Code profile label is not a security boundary. Crossing environment state or
silently reusing credentials fails closed.

## Dependency, startup, and shutdown order

Startup verifies Windows and storage health, approved identity and private-route
availability, profile isolation, then VS Code compatibility and the integrity
and version of the locally retained Glass Codex package. Only afterward may the
client authenticate to governed Cloud Atlas APIs. Harmony, Coppermind,
Emberdark, Phoenix, Notum/Sentinel, Forge, backups, and Prime recovery never
start through or depend on Apollo.

Shutdown first stops interactive jobs and sessions, records or transfers any
approved work, revokes ephemeral access, and cleans or quarantines protected
caches. It then closes Glass Codex/VS Code and Windows. Persistent Cloud Atlas,
monitoring, backup, Forge, media, and recovery services continue independently.

## Source, backup, restore, and reinstall continuity

Apollo holds no unique protected original, Mission state, Coppermind state,
credential, canonical repository, unpushed commit, unique branch, or sole
working copy. Source continuity requires clean or truthfully dispositioned
worktrees, pushed reviewed refs or another approved preservation route, and a
representative clean-clone Prime validation from an independent endpoint.

Retain an integrity-verified, versioned Glass Codex installer and rollback
package; Windows recovery-media disposition; manufacturer-driver source; and
an approved system-image disposition. Do not back up live credentials,
unmanaged protected caches, raw private logs, or unique canonical state as
ordinary endpoint artifacts.

Reinstall verifies Windows and driver health, restores no secret from ordinary
backup, reinstalls the verified known-good client package, recreates the three
isolated environments, obtains fresh bounded credentials, confirms cleanup,
and proves one native Windows-specific workflow plus one Glass read workflow.
Hermes separately proves a portable command, administration, and clean-clone
recovery workflow.

## Degraded states and rollback

Apollo loss removes the primary GUI and Windows work surface only. Remote-route
loss falls back to direct local Apollo use or independent Hermes administration;
it never blocks Prime recovery. Glass failure disables or rolls back the client
without reverting backend state. Unavailable Windows recovery, Hermes
independence, route security, package integrity, cache isolation, or source
continuity is a blocking proof state, not permission to proceed.

## Proof and stop boundary

Required protected execution evidence includes native VS Code and a
representative Windows-specific workflow on Apollo, verified package install,
approved private access from the bounded clients, environment-isolation and
cleanup denial tests, restart/loss proof, Hermes independent recovery, source
inventory/disposition, and Windows recovery posture.

Mission #285 supplies architecture and the exact future proof plan only.
Commissioning, installation, credentials, remote access, migration, device
rename, Windows removal, Proxmox work, and PF-C01-M02 exit remain
`PENDING_PROTECTED_EXECUTE`; no runtime acceptance is claimed.
