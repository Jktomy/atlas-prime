# Deterministic conservation and generated checkpoints

Former G4-E/G4-F intent is conserved here without restoring obsolete phase
authority. Former G4-E means only the construction layer: a read-only package
preparer, exact Ubuntu/Windows reconciliation, and one narrow
`GENERATED_CHECKPOINT_V1` construction profile that invokes the singular Thread
Engine and stops at draft-PR readback. Former G4-F means only the later live
acceptance layer and requires its own exact-head evidence transaction. Neither
label is an active phase, authorizer, writer, or promotion mechanism.

The conserved controls are deterministic projections, bounded publication, exact source,
candidate, workflow, parity, and receipt binding, cross-platform byte
reproduction, replay and stale-base rejection, evidence conservation, and
authored/generated separation.

The five approved `generated/` reports remain non-governing. Ubuntu and Windows
must independently build the same source fingerprint and byte-identical ordered
file SHA-256 register. Reconciliation binds both artifact digests to one exact
mission, canonical base, workflow source, run, and attempt before publication.

The owner-triggered hosted route:

1. accepts one fresh mission ID and replay nonce for one exact canonical `main`;
2. requires both event actor and triggering actor to equal the repository owner;
3. requires the workflow source SHA to equal that base;
4. verifies the reconciled parity register against the bytes it will publish;
5. constructs an exact mission package without invoking Git or GitHub;
6. invokes only the singular Thread Engine, which rechecks remote `main`,
   rejects any pre-existing deterministic branch or PR, permits only the five
   approved generated paths, and creates one immutable replay-bound branch and
   one draft PR;
7. performs exact PR and file-set readback, writes a sanitized immutable
   receipt outside the repository, records
   `DRAFT_CREATED / REQUIRED_PR_VALIDATION_PENDING`, and stops; and
8. relies on the normal Prime pull-request workflow as the single authoritative
   candidate-validation layer for the generated head, including required
   `prime/integrity` and conditional `prime/windows-compatibility`.

The publisher does not run a second exact-head Ubuntu/Windows validation matrix.
Parity proof remains inside the publisher, candidate validation remains inside
the ruleset-governed pull-request path, and the write-capable publication job
remains permission-isolated from read-only admission, parity, reconciliation,
and preparation.

The route and profile cannot ready, merge, force-push, write `main`, change settings,
advance a Quest, promote a capability, or become a second writer. Rejection is
pre-mutation. Any ambiguous result after branch push is `PARTIAL`, preserves the
remote state, emits sanitized readback evidence when possible, and forbids an
automatic retry. Recovery uses a new reviewed mission after reconciliation.

Generated parity and an ordinary green CI run do not alone prove AJ-09. AJ-09
requires one live owner-triggered workflow whose hosted Ubuntu/Windows artifacts
are byte-identical and whose hosted route itself causes the singular Thread
Engine to create the exact-base generated-only draft PR and receipt. Capability and Campaign promotion occur
only in a later authored evidence transaction after exact-head CI, detached
review, PR readback, and canonical readback.
